# 白易 FB (Fast Binary) 封包框架升级与修复总结

## 1. 内存崩溃 Bug 修复 (StrW / StrA / Bytes 解包)
- **问题现象**：执行 `_deFB_get` 解包变长字符串/字节集时，引发 `0xC0000005` 内存写入访问违规。
- **根源分析**：原代码使用了 `out.bytes = Bytes((int)sz);`。因为 `Bytes` 的构造函数是 `explicit` 的临时内存分配，通过 `=` 赋值给外部变量时，底层没有妥善完成右值移动，导致缓冲区指针成了 `NULL`，随后执行的 `memcpy(out.bytes.buf, p, sz)` 直接踩空崩溃。
- **解决方案**：全面改用深拷贝与原生安全构造机制。
  - `StrW` / `StrA` 改用原生反射安全函数：`out = ToStr<W>(BR(p, sz));`
  - `Bytes` 解包改用标准双参构造拷贝：`out = Bytes(p, sz);`
  - 彻底杜绝野指针赋值，解包过程绝对安全。

## 2. 完美打通多维/嵌套数组 (Arraybe<Arraybe<T>>)
- **问题现象**：在支持 `超级列表框` 序列化时，传入了真正的二维变长数组 `Arraybe<Arraybe<StrW>>`。封包虽然成功，但解包读取时必崩。
- **根源分析**：
  - 之前的 `FB` 框架其实只能序列化“一维变长”，并不支持真正的“多维/嵌套变长”。
  - 当底层 `_FB_Array` 在打包时，如果遇到的元素仍然是 `Arraybe` 类型，它错误地调用了最底层的强制取址函数 `FBRef(aa[i])`。
  - 由于没有专门重载，`FBRef` 触发了通用的内存快照逻辑 `BR(&a, sizeof(T))`，直接把 `Arraybe` 类本身所在的堆栈内存（包含原始的野指针与计数值）当成了普通结构体浅拷贝进了包里！
  - 解包时，这些脏指针被读取出来当成合法的包内存指针，导致寻址全面越界。
- **解决方案**：
  - 引入了递归思维：将 `FB.h` 和 `FB.cpp` 的序列化信道打通。
  - `_FB_Array` 检测到子元素仍是变长对象时，不再调用简易的 `FBRef`，而是改用并列推导的 `build_FBPart(aa[i])`。
  - 如此一来，每一个子数组都会被彻底展开为独立的标准 FB 子包并妥善托管内存。现在框架**完美支持了任意层级深度的无限嵌套序列化**。

## 3. 架构精简与底层优化
- **移除冗余的包壳**：彻底删除了原有的过渡用结构体 `FBPart`。这得益于发现底层 `Bytes` 实际上已天然支持 `BRef` 弱引用构造机制，直接利用无拷贝特性进行生命周期管理，减少了结构体定义。
- **接口大一统**：去除了早期为了单个参数特意保留的单参重载版 `FB(const T& a)`，强制统一收口到基于 C++17 特性的变参模板 `FB(const T&... aa)` 中，大幅精简了系统库代码。
- **底层方法隐藏**：将解包所需的重载反射推导支撑函数（如 `deFB_get`、`deFB_impl`）全面加上下划线隐藏为内部私有函数 `_deFB_get`。让外界调用智能提示回归到极致极简的 `FB()` 和 `deFB()` 身上。
